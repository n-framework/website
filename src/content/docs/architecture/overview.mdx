NFramework enforces Clean Architecture through project structure, package boundaries, and automated checks. This guide explains the layer model, folder layout, dependency direction, and core design principles.

## Workspace Structure

A workspace is the root container for one or more services. Created with `nfw new <name>`, it has a simple layout:

```
my-workspace/
├── nfw.yaml          # Workspace metadata (name, template, namespace, services)
├── README.md         # Workspace documentation
└── src/              # Service directory — each service lives here
    └── <service>/    # Generated by nfw add service
```

The `nfw.yaml` file tracks workspace identity and every service added to it:

```yaml
workspace:
  name: BillingPlatform
  template: official/blank-workspace
  namespace: BillingPlatform

services:
  Orders:
    path: src/Orders
    template:
      id: official/dotnet-service
      version: 1.0.0
```

### Folder Conventions

| Folder            | Purpose                                                                          |
| ----------------- | -------------------------------------------------------------------------------- |
| `core/`           | Domain and Application layers — pure business logic                              |
| `infrastructure/` | Infrastructure adapters — persistence, external services                         |
| `presentation/`   | API layer — HTTP endpoints and hosting                                           |
| `Features/`       | Feature-focused code within each layer (entities, commands, handlers, endpoints) |
| `Shared/`         | Cross-cutting code within a layer (registration, configuration, utilities)       |

## Layer Responsibilities

### Domain (`core/<ServiceName>.Domain/`)

The Domain layer contains the core business model:

- **Entities** with strongly-typed identities
- **Aggregate Roots** that enforce invariants
- **Value Objects** for immutable domain concepts
- **Domain Events** for intra-aggregate communication

The Domain layer has **zero external dependencies**. It depends on nothing but the language runtime.

### Application (`core/<ServiceName>.Application/`)

The Application layer orchestrates business workflows:

- **Commands** and **Queries** (CQRS pattern)
- **Handlers** that implement use case logic
- **Result types** for explicit outcome handling
- **Abstractions** for persistence, validation, security, and messaging
- **`ApplicationServiceRegistration.cs`** — DI registration for application services

Application code depends on the Domain layer and framework abstractions only. It never references infrastructure libraries directly.

### Infrastructure (`infrastructure/<ServiceName>.Persistence/`)

The Infrastructure layer implements framework abstractions using concrete libraries:

- **Persistence adapters** (e.g., EF Core DbContext, repositories)
- **Validation adapters** (e.g., FluentValidation)
- **Logging adapters** (e.g., Serilog)
- **Security adapters** (e.g., JWT authentication)
- **`InfrastructurePersistenceRegistrationExtensions.cs`** — DI registration for infrastructure services

Infrastructure depends on Application abstractions, never the reverse.

### API (`presentation/<ServiceName>.WebApi/`)

The API layer exposes HTTP endpoints:

- **`Program.cs`** — application entry point and host configuration
- **Minimal API** route definitions
- **Generated routes** from commands and queries
- **Middleware** for exception handling, problem details, and localization
- **Health check** endpoints (`Shared/HealthCheck/`)
- **OpenAPI/Swagger** configuration (`Shared/OpenApi/`)

The API layer depends on Application and Infrastructure. It never contains business logic.

## Dependency Rules

```
Api → Application, Infrastructure
Infrastructure → Application, Domain
Application → Domain
Domain → (nothing)
```

**Forbidden dependencies:**

| Layer          | Must NOT Depend On                   |
| -------------- | ------------------------------------ |
| Domain         | Application, Infrastructure, Api     |
| Application    | Infrastructure, Api                  |
| Infrastructure | Api                                  |
| Api            | Domain (must go through Application) |

The `nfw check` command validates these rules automatically.

## Namespace Convention

Namespaces follow the pattern `<WorkspaceNamespace>.<ServiceName>.<Layer>`:

```
BillingPlatform.Orders.Domain
BillingPlatform.Orders.Application
BillingPlatform.Orders.Infrastructure.Persistence
BillingPlatform.Orders.WebApi
```

## Feature Folder Pattern

Each layer uses a `Features/` directory for feature-focused organization:

```
core/Orders.Domain/Features/Orders/
├── Order.cs                    # Entity
├── OrderCreatedEvent.cs        # Domain event
└── OrderId.cs                  # Strongly-typed ID

core/Orders.Application/Features/Orders/Commands/CreateOrder/
├── CreateOrderCommand.cs
├── CreateOrderCommandHandler.cs
└── CreateOrderCommandValidator.cs

infrastructure/Orders.Persistence/Features/Orders/
├── OrderConfiguration.cs       # EF Core entity configuration
└── OrderRepository.cs

presentation/Orders.WebApi/Features/Orders/
└── OrderEndpoints.cs           # Minimal API route definitions
```

## Core Design Principles

### Clean Architecture by Default

Layer boundaries, dependency direction, naming conventions, and generated code structure all reinforce separation of concerns. The `nfw check` command enforces these rules automatically.

### Compile-Time Over Runtime

Framework behavior such as DI registration and route discovery is generated at build time. No runtime assembly scanning is required for generated registrations.

### Explicit Results Over Exceptions

Application workflows use `Result<T>` and explicit outcome types for expected business outcomes. Validation failures and business rule violations are represented as data, not exceptions.

### Zero-Dependency Core

Framework packages have no external library dependencies. Third-party integrations live in separate adapter packages that depend on framework abstractions.

### Replaceable Adapters

Swapping one adapter for another (e.g., EF Core to Dapper) requires no changes to Domain or Application code. Library-specific types remain isolated to adapters.

## CLI Architecture

The `nfw` CLI itself follows the same layered approach:

```
src/nframework-nfw/
├── core/
│   ├── nframework-nfw-domain/         # Domain models for CLI operations
│   └── nframework-nfw-application/    # Command handlers and business logic
├── infrastructure/
│   ├── nframework-nfw-infrastructure-filesystem/
│   ├── nframework-nfw-infrastructure-git/
│   ├── nframework-nfw-infrastructure-yaml/
│   └── nframework-nfw-infrastructure-versioning/
└── presentation/
    └── nframework-nfw-cli/            # CLI commands and runtime
```

The CLI uses a specification-based command model (`CliSpec`, `CliCommandSpec`, `CliOptionSpec`) with a clap-backed runtime that dispatches to registered handlers by canonical command key (e.g., `templates/list`, `add/service`).

## Validation

Run `nfw check` to validate architecture boundaries in any workspace. The command:

- Scans for forbidden project references
- Detects namespace or package violations
- Exits with non-zero status on failure
- Provides actionable remediation hints
- Runs in CI without interactive input

Use `nfw check --verbose` for detailed diagnostics including the specific file or dependency that violates a rule.
